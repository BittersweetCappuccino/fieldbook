Pressure Test: Fieldbook's Sync Feed vs. a Real OEM API

Addendum to the Fieldbook — Dealer Operations case study.

The original Fieldbook concept closed with a commitment: before treating the manufacturer sync feed as a real design, pressure-test it against an actual OEM's API constraints. This addendum does that, using John Deere's public developer portal (developer.deere.com) as the reference contract — Deere being both the most prominent OEM in the mockup and the one with the most mature public documentation.

The short version: the concept's mental model of the OEM relationship was too optimistic in three specific ways, and each one changes a product decision.

What the mockup assumed

The Fieldbook dashboard renders the manufacturer feed as a stream of discrete, timely events — a warranty credit posts, a shipment ETA updates, a claim gets flagged — implying that OEM data arrives in near-real-time, that integrated entities are stable references, and that "integrating with an OEM" is a single connection with uniform data behind it.

What the documentation actually says

1. The dominant pattern is polling, not push.
Deere's docs describe an eTag mechanism for clients that poll APIs on a regular cadence, returning only data modified since the last request. The existence of purpose-built polling optimization tells you what the architecture expects: the client asks repeatedly; the OEM does not push. A "live feed" is really a "recently polled feed."

2. The OEM will throttle you, and the contract says how.
Rate limiting is explicit: overwhelmed servers return HTTP 429 with a Retry-After header, and clients are required to wait out the specified interval before retrying. The docs even distinguish between synchronous flows (where a human is waiting and the app should error out) and asynchronous flows (where the app should always honor the wait). Backpressure isn't an edge case — it's a documented, expected state.

3. Cached OEM data goes stale by design.
Resource IDs are meant to be parsed from fresh API responses immediately before use, because access to a resource can change or be revoked over time. A DMS cannot mirror OEM entities into its own database and treat them as durable — the OEM's data model assumes re-verification.

4. "One integration" is actually a bundle of licensed scopes.
Access is OAuth 2.0 with per-endpoint scopes, granted explicitly by the authenticating user, and an application can be denied at the endpoint level if it isn't licensed for it. Two dealers "integrated with John Deere" may have materially different data available to them depending on which scopes were granted and which endpoints the application is licensed for.

What changes in the product

Concept assumption Documented reality Product decision
Feed events arrive in real time Polling cadence + eTags; freshness is bounded by poll interval Every OEM panel shows a "last synced" timestamp; the UI never implies liveness it can't guarantee
Sync failures are exceptional 429 + Retry-After is a normal, documented state Throttling surfaces as a calm "sync delayed — retrying at HH:MM" status, not an error banner; the dealer sees a schedule, not a failure
OEM entities can be cached as stable records Resource references expire; access can be revoked The system of record stores our work orders and parts records with OEM references, re-resolved at read time — never mirrored OEM state treated as truth
One OEM = one integration Per-endpoint licensing and user-granted scopes Integration status is displayed per capability (warranty claims ✓, parts availability ✓, financial data —), so a dealer knows exactly what their connection covers

Why this matters beyond Deere

Deere happens to publish its contract; Kubota, Case IH, and Toro each have their own, with different maturity levels and different gaps. The design consequence is that the integration layer can't be built to the best OEM's behavior — it has to be built to a capability model that degrades gracefully: each OEM slot in the feed declares what it supports (push vs. poll, event types, freshness guarantees), and the UI renders honestly against that declaration.

That's the real lesson of the pressure test. The first version of the sync feed was designed around what a dealer wants to see. The revised model is designed around what the product can truthfully promise — and makes the gap between the two visible instead of papering over it.

Method note

This analysis was done entirely from public developer documentation — no API access required. For a production effort, the next steps would be: sandbox access to validate assumed payload shapes against real responses, a comparison pass across the other three OEMs' partner programs to populate the capability model, and a conversation with an actual dealer principal about which sync delays are tolerable and which block revenue.
